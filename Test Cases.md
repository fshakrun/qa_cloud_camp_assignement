# Процесс тестирования нового функционала
Для проверки тестирования нового функционала, предполагающего возможность самостоятельно развернуть базу данных PostgreSQL, исходя из [озвученных в задании требований](https://github.com/QACloudCamp/test-assignment), предлагаю нижеследующую стратегию тестирования. 
Данная стратегия учитывает основные варианты использования функционала и позволяет выявить потенциальные риски.

## Стратегия тестирования (применяемые виды тестирования)

1. Smoke-тесты, выявляющие степень готовности функционала и возможность дальнейшего тестирования.
Проверки предполагают выявление доступности сервиса, наличие необходимой формы в веб-интерфейсе сервиса и кнопки "Создать" (базу данных).

2. Функциональное тестирование — то есть проверки работоспособности нового функционала сообразно требованиям к нему и общей логики взаимодействия конечного пользователя с ним.
С помощью техники граничных значений и классов эквивалентности осуществляем проверки корректности работы полей ввода данных: "Название БД", "Регион размещения", "Размер БД".

3. Кроссбраузерное/Кроссплатформенное тестирование — в зависимости от озвученных требований осуществляем проверку работы функционала в различных браузерах и опериационных системах.

4. Регрессионное тестирование — для проверки корректности работы функционирования сервиса после добавления нового функционала.
